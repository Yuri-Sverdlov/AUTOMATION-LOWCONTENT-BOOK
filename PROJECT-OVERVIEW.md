---

## 1. Идея проекта

Конвейер для производства **low-content книг** под Amazon KDP (блокноты,
планировщики, журналы с промптами, судоку, раскраски, логбуки и т.п.).
Цель — автоматизировать кодом всё «тяжёлое» производство, оставив человеку
только 10 минут в день на ручную загрузку готовых файлов в KDP.

Принцип: **код — мастерская, которая готовит холсты и краски; финальный мазок
на «холсте Amazon» кладёт только рука человека.**

---

## 2. Граница безопасности KDP (красная линия)

Это самое важное ограничение — нарушение грозит баном аккаунта.

**Запрещено и ведёт к бану:** автоматизация браузера (Playwright, Selenium,
Puppeteer), screen scraping, недокументированные API. Amazon с марта 2026
прямо прописал это в политике и детектирует по сотням сигналов (тайминги,
CDP-протокол и т.д.) — скрыть практически невозможно. Санкции по нарастающей:
suppression (книги пропадают из поиска) → приостановка → пожизненный бан.

**Безопасная граница:** код доходит ровно до момента *перед* веб-формой KDP и
останавливается. Он готовит файлы (PDF интерьера, PNG обложки) и шпаргалку со
всеми значениями полей. Дальше — человек: открываешь KDP в обычном браузере и
для каждого поля делаешь «выделить в шпаргалке → копировать → вставить»,
файлы подгружаешь кнопкой. ~2-3 минуты на книгу. Для Amazon это обычный
человек, печатающий текст.

**Прочие триггеры бана, которых избегаем:**
- дубль-контент (одинаковый интерьер/метаданные в разных книгах);
- более 3 книг в сутки → аккаунт флагируется как abusive (жёсткий лимит);
- AI-контент без декларации (с 2026 обязательна).

## 3. Полный цикл производства

### Фаза 0 — Паспорт ниши (контракт)
Перед генерацией разведка выдаёт `niche.yaml`: тип интерьера, формат, палитра
из топа, шрифты, формула названия, 7 backend-ключей, шаблон описания, цена.
Весь конвейер кормится из этого одного файла.

### Фаза 1 — Нишевая разведка (раз в 3-4 недели)
Отвечаем на 4 вопроса:
- **Спрос:** BSR топа-10 в категории ниже ~100 000 → ниша живая.
- **Конкуренция:** мало рецензий (< ~50) при высоком рейтинге → окно.
- **Почему в топе:** разбираем обложки (палитра, шрифт, мини­мализм vs
  иллюстрация), структуру описания, формулу названия
  (`[аудитория] + [тип] + [уникальный элемент]`), backend-ключи.
- **Насыщенность дублями:** если топ забит почти одинаковыми книгами — ниша
  перегрета ботами, не входим (и риск флага за дубль).

На выходе — 2-3 ниши на месяц, оформленные как паспорта.

### Фаза 2 — Движок серии (1-2 дня)
Не «шаблон на 20 книг», а **параметрический движок**: один генератор
интерьера принимает параметры из паспорта и выдаёт сколько угодно валидных
PDF; то же для обложек. Движок переживает смену ниши — шаблон нет.

### Фаза 3 — Ежедневный конвейер (3 книги/день × 30 дней)
Три книги дня = один тип интерьера × три разных обложки/угла/набора ключей
(не три разные концепции). Между днями меняем микро-параметры, чтобы 30 книг
месяца не были байт-в-байт одинаковыми.
- **Идемпотентность:** ведём `published.csv`, сверяемся, чтобы не выпустить
  дубль.
- Утро: скрипт генерит 3 комплекта (interior.pdf + cover.png + crib.md).
- День: вручную грузишь 3 книги в KDP (~10 мин).

### Фаза 4 — Петля обратной связи (раз в неделю)
Записываем BSR каждой книги в журнал. «Сдвинулась вверх» = ранг уменьшился
(напр. #800 000 → #150 000) = пошли продажи. Победившие паттерны (формула
названия, стиль обложки, ключи) переносим в паспорт следующей ниши. Система
учится на фактах, а не на интуиции.

## 4. На что смотреть при поиске тренда

- **BSR (Best Sellers Rank):** меньше число = лучше продажи. Главный сигнал
  спроса и единственный объективный критерий «выстрелила ли книга».
- **Число и свежесть рецензий:** мало рецензий + высокий рейтинг = молодое
  незанятое окно.
- **Разнообразие топа:** разнородный топ = есть место; клоны = перегрев.
- **Сезонность:** Google Trends — чтобы ловить нишу до пика, а не после.
- **Типы low-content для анализа** (помимо линованных блокнотов): судоку и
  сканворды, раскраски (мандалы, животные, паттерны), планировщики (daily,
  habit, budget), журналы с промптами (gratitude, prayer, dream), логбуки
  (reading, fitness, food, baby), скетчбуки, дот-грид, гостевые книги,
  книги рецептов с пустыми страницами, детские активити-буки.

## 5. SEO-ядро

- **Источник ключей:** Amazon autocomplete (бесплатно; вводишь
  «gratitude journal for» и собираешь подсказки — именно оттуда берут данные
  платные сервисы).
- **Анализ топ-10 конкурентов:** восстанавливаем рабочие ключи и формулы.
- **7 backend-слотов KDP** по 50 символов — заполняем без повторов между
  книгами одной серии (небольшие вариации).
- **Описание в HTML:** KDP принимает `<b>`, `<i>`, списки и т.д. Размеченное
  описание читается лучше «простыни» → выше конверсия. Поэтому шпаргалка
  хранит описание сразу в HTML.
- **Название по формуле ниши:** `[аудитория] + [тип] + [крючок]`.

## 6. Платное vs бесплатное (правило)

> **Покупай данные, которые Amazon не отдаёт легально. Пиши код, который
> превращает данные в продукт.**

**Покупать (это данные):** реальные BSR, оценки спроса и конкуренции. Amazon
не даёт публичного API, scraping запрещён. Здесь оправданы
**Publisher Rocket** (разовый ~$97) или **Book Bolt** (подписка). Это
инвестиция в Фазу 1.

**Писать самому (превращение данных в продукт):** движок интерьеров,
генератор обложек (Pillow), сборка SEO-строк, генерация названий/описаний
через LLM (Claude API или локальная модель через Ollama), шпаргалка, журнал.
Ни один сервис не делает это под твой паспорт и твой темп 3/день одним
запуском. Здесь — независимость и преимущество.

**Бесплатные заменители (если бюджет в ноль):** Amazon autocomplete (ключи),
Google Trends (сезонность), ручной разбор топ-20 раз в месяц — ~80% пользы
платного инструмента, но медленнее и без точных BSR.

**Вердикт:** один платный инструмент разведки + полностью свой
производственный код. Сервисы-комбайны (Book Bolt умеет интерьеры) привязывают
к чужим шаблонам и убивают автогенерацию серии из 30 вариантов.

## 7. Технологический стек (предварительно)

- **Интерьер:** `reportlab` / `fpdf2` (генерация PDF по формату).
- **Обложка:** `Pillow` (PIL) + шрифты + палитры из топа.
- **LLM:** Claude API или Ollama (названия, описания).
- **Данные:** Publisher Rocket / Book Bolt (разведка), Amazon autocomplete,
  Google Trends.
- **Язык:** Python.

---

## 8. Следующие шаги

1. Спроектировать формат `niche.yaml` (паспорт ниши) детально.
2. Написать движок интерьера для одного типа (напр. prompt-journal).
3. Написать генератор обложек под палитру/шрифты паспорта.
4. Собрать генератор шпаргалки (crib.md) с описанием в HTML.
5. Завести `published.csv` и процедуру еженедельного анализа BSR.

> Заметка по модели: проектирование/стратегию удобно вести на Opus, рутинное
> написание кода — на Sonnet (расходует лимит подписки ~5× экономнее).
