# BACKLOG: ввод параметров — конфиг-файл и/или браузерный UI

**Создано:** 2026-06-08 (архитектор) · **Статус:** идея, не активно
**Активировать:** перенести в `tasks/TASK.md`, когда дойдёт очередь.

---

## Идея пользователя

Нужен удобный способ заполнять важные параметры запуска (размер книги, число
страниц, trim, бумага, заголовок, слаг ниши, путь к арту и т.п.) — **конфиг-файл
или, лучше, UI в браузере** — вместо передачи аргументов в коде/CLI.

## Зачем

Сейчас параметры зашиты в CLI/вызовах движков. Человеку нужно менять их без правки
кода. Конфиг = единая точка правды на запуск; UI = ещё и без редактирования текста руками.

## Два уровня (можно делать поэтапно)

**Уровень 1 — конфиг-файл (дёшево, делать первым).**
- Один файл на запуск (YAML), напр. `data/niches/<ниша>/niche.yaml` УЖЕ частично это
  (см. STRUCTURE.md). Расширить его до полного «паспорта запуска»: trim, pages, paper,
  bg_color, title, путь к front_png.
- `build_cover.py` и остальные движки читают параметры из этого конфига (CLI-аргументы
  остаются как override).
- Критерий: сменить книгу = поменять один YAML, код не трогается.

**Уровень 2 — браузерный UI (дороже, после уровня 1).**
- Локальная страница (форма) → заполняешь поля → жмёшь «собрать» → бэкенд дёргает
  `build_cover.py` и кладёт результат в дерево ниши.
- Технически: лёгкий локальный сервер (FastAPI/Flask) + простая HTML-форма, ЛИБО
  статическая форма, пишущая `niche.yaml`, который потом читает CLI.
- **Красная линия:** UI локальный, против KDP ничего не автоматизирует (см. AGENTS.md).

## Связи

- Конфиг — надстройка над контрактом `niche.yaml` (STRUCTURE.md).
- UI уровня 2 имеет смысл, когда устаканится набор параметров уровня 1.

## Открытые вопросы

- Делаем оба уровня или достаточно конфига уровня 1?
- Где хранить конфиг запуска: расширять `niche.yaml` или отдельный `run.yaml` на книгу?
- Стек для UI (FastAPI+HTML / Flask / Streamlit).
