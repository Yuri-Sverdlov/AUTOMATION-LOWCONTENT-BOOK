## Цель пользователя (его слова)

Запустить алгоритм и получить «на склад» ~**12 названий книг на ближайшие 4 дня**.
Вход: пользователь РУКАМИ выбирает на Amazon 2 фразы-крючка из топа (легально, без скрейпинга)
и передаёт их генераторам.

## Архитектура воронки (5 ступеней пользователя)

1. **Генерация:** 3 агента («Шутник-1/2/3»), каждый на входе 2 крючка → выдаёт **6–7 вариантов**
   названий. Итого ~18–21 кандидат.
2. **Отбор/оценка:** те же 3 агента, но с другими системными промптами выступают как
   **«Арбитр-1/2/3»**.
3. **Список:** все кандидаты сводятся в один список.
4. **Отбор по критериям:** жёсткие фильтры (safety, длина KDP, ≤7 ключей).
5. **Баллы:** каждый арбитр ставит 1–10 по критериям; **индекс = сумма баллов арбитров**;
   список ранжируется по индексу.

## Два варианта реализации (можно МИКСОВАТЬ по ступеням)

- **V1 — одна мультимодальная LLM** (напр. opus 4.6) пишет 3 разных системных промпта-стиля
  → 3 «личности». Дёшево, стабильно, но голос модели один → варианты коррелируют.
- **V2 — три разные модели** (через API или локально). Настоящее разнообразие, дороже/сложнее.
- **Решение пока:** на ГЕНЕРАЦИИ важно разнообразие → ближе к V2 (или явно разные стили +
  высокая температура); на ОЦЕНКЕ важнее единая шкала → можно одна модель в роли 3 арбитров.

## Антиперекос судейства

- **Round-robin против самооценки:** Арбитр-N НЕ оценивает варианты своего же автора (Шутник-N).
  Тогда каждый кандидат судят ровно **2 арбитра** (делитель = 2, а не 3) — честно и сравнимо.
- **Две схемы свёртки баллов (реализовать ОБЕ, чтобы сравнить ранжирование):**
  - **A. Сырая сумма** — просто складываем все баллы.
  - **B. z-score** — у каждого арбитра `z=(балл−его среднее)/его разброс`, потом сумма z.
    Убирает перекос «строгий vs добрый судья». Нужно ≥6–7 кандидатов на судью (у нас есть).
    Делитель дисперсии (n или n−1) выбрать один и применять ко всем судьям.

## Принятые решения (2026-06-11, сессия 11) — закрывают 7 открытых вопросов

**в6 — Модели.** Делаем **Вариант А**, **Вариант B** закладываем в конфиг на будущее.
- **А (сейчас):** ОДНА модель и на генерации, и на оценке; разница — только системные
  промпты-личности + температура. Шутникам — высокая температура (можно разную: 0.8/0.9/1.0);
  арбитрам — низкая ненулевая (~0.2–0.3).
- **B (на будущее):** на генерации — разные модели (шутники = разные провайдеры/модели),
  на оценке — одна модель в 3 ролях. Опционально часть характеристик тоже разная.
- **Условие перехода А→B:** модель/температура/системный промпт КАЖДОГО агента задаются
  в конфиге (`config/models.yaml`, режим `advanced` поддерживает per-role словарь), а не в
  коде. Тогда B = правка конфига без переписывания кода.
- **Известные ограничения А (причина существования B):** (1) одна модель на генерации →
  варианты коррелируют, склад однообразнее; (2) генератор = судья (общая подложка-модель) →
  остаточный самосуд, round-robin гасит его лишь на уровне личностей, не модели.

**в1+в7 — Свёртка баллов.** Реализовать ОБЕ схемы (сырая взвешенная сумма и z-score).
Веса — рубрика DOE (hook_strength 0.30 …). **Дефолт = z-score поверх весов**: сначала
z-нормализация баллов по каждому судье, потом взвешенная сумма по критериям.
См. «Приложение: как работает z-score».

**в2+в3+в4 — Жёсткие фильтры / критерии.**
- шлагбаум `safety_risk < 7 → выброс` — **ДА** (анти-бан);
- шлагбаум «длина названия > лимита KDP → выброс» — **ДА** (отдельный хард-гейт, не балл);
- `audience_fit` — брать аудиторию из `niche.yaml`, НЕ фиксировать 20–45 (это правка
  критерия оценки, а не фильтр).

**в5 — Наборы крючков на запуск.** Число наборов крючков на прогон — **ПАРАМЕТР конфига
(1..N)**. «Набор» = крючки, отдаваемые шутникам разом (по умолчанию 2 фразы, но число не
свято и связь между ними не требуется). Математика генератора неизменна: набор → 6–7
вариантов на шутника.

**Round-robin (уточнение).** Арбитр-N не судит шутника-N → каждый кандидат оценивают 2
судьи (делитель 2). В Варианте А смысл round-robin — варьировать набор сравнения; в B —
ещё и анти-самосуд (там судья ≠ генератор по модели).

## Приложение: как работает z-score (памятка для реализации)

> Перенесено из обсуждения сессии 8, чтобы не потерять при коммите. Это пояснение
> математики свёртки баллов — зачем нужна стандартизация и как её считать.

**Проблема, которую лечит z-score.** Один арбитр строгий (ставит 17–19), другой щедрый
(ставит 12–26). Простая сумма даёт перекос: большие числа щедрого судьи перетягивают
ранжирование на себя. Деление суммы на константу (среднее) НЕ помогает — порядок в списке
не меняется, лечится только «масштаб числа», а не «характер судьи».

**Формула.** Для каждого арбитра отдельно: `z = (x − μ) / σ`, где `x` — балл, `μ` — средний
балл этого арбитра по всем вариантам, `σ` — его стандартное отклонение. Вычитание `μ` убирает
сдвиг (строгий и щедрый центрируются в нуле); деление на `σ` уравнивает разброс шкал. Итог:
сравниваем не сырые баллы, а «насколько вариант выделяется на фоне привычек именно этого судьи».
Итоговый z-индекс = сумма z по всем судьям, оценивавшим этот вариант.

**Мини-пример (3 названия, 2 арбитра, по строковой сумме баллов):**

| Название | Σ Арбитр 1 (щедрый) | Σ Арбитр 2 (строгий) | Простая сумма | Σ z |
|---|---|---|---|---|
| Берёзка | 19 | 19 | 38 | **+1.22** |
| Дубок | 12 | 18 | 30 | −1.22 |
| Рябина | 26 | 17 | 43 | 0.00 |

Простая сумма ставит 1-м Рябину (43) — но только потому, что щедрый судья влепил ей 26,
а строгий поставил её последней (17). z-score переворачивает ранг → 1-я Берёзка: голоса
по Рябине погасили друг друга (+1.22 и −1.22), а Берёзка была фаворитом строгого судьи.

**Как считается σ (разброс), на примере Арбитра 1 — баллы 19, 12, 26:**
1. среднее `μ` = (19+12+26)/3 = 19;
2. отклонения: 0, −7, +7;
3. квадраты: 0, 49, 49;
4. среднее квадратов = 98/3 ≈ 32.67; `σ` = √32.67 ≈ **5.72**.

**Нюанс делителя дисперсии.** Шаг 4 делит сумму квадратов на `n` («популяционное» σ,
так делает `numpy.std()`) либо на `n−1` («выборочное», так делает `pandas.std()`). Для нашей
задачи годится любой — главное выбрать ОДИН и применять ко всем судьям одинаково.

**На Python** весь расчёт — пара строк: `z = (scores - scores.mean()) / scores.std()` для
каждого арбитра, потом сумма z по судьям. z-score осмыслен при ≥6–7 вариантах на судью
(у нас 18–21 на пару крючков — нормально); на 2–3 кандидатах он шумный.

**Аналогия (живопись):** два преподавателя оценивают этюды. Один — сухарь, выше четвёрки
не ставит; другой всем рисует пятёрки. Сравнивать сырые оценки нечестно. z-score спрашивает
каждого: «насколько эта работа выше/ниже твоей обычной планки?» — и делает голоса
сопоставимыми. `σ` здесь — как широко мазки разлетелись вокруг центральной точки (среднего).
