# BACKLOG: палитровая автоматизация арта (генерация фронтов «отличаются только цветом»)

**Создано:** 2026-06-08 (архитектор) · **Статус:** идея, не активно
**Активировать:** перенести в `tasks/TASK.md`, когда дойдёт очередь.

## Идея пользователя (дословный смысл)

Есть фиксированный **фон** и **3–4 фиксированных дизайнерских элемента**. Элементы
накладываются на фон **слоями**, причём каждый слой должен быть **перекрашиваемым**.
На вход даётся **10–12 палитр**. Скрипт генерирует набор вариантов фронта, где
**каждый вариант отличается ОТ другого ТОЛЬКО цветовой палитрой** (композиция,
позиции, формы — идентичны).

Аналогия: один и тот же трафарет-плакат, который прокатывают через 12 разных наборов
краски. Рисунок один, меняется только цвет.

## Зачем

Дешёвый способ получить визуально разные обложки одной серии без перерисовки арта —
массовое разнообразие для KDP при сохранении узнаваемого «скелета» серии.

## Эскиз контракта (обсудить при активации)

- **Слои:** фон + N элементов как раздельные перекрашиваемые слои. Формат хранения
  слоёв обсудить: PNG с альфой (маска→заливка) vs SVG (заливка по id) vs параметрические
  фигуры. SVG даёт «чистую» перекраску по `fill`; PNG-маски проще, но требуют
  аккуратной альфы.
- **Палитра:** список палитр (по 3–5 цветов) на вход (YAML/JSON). Привязка «слой → роль
  цвета в палитре» (напр. слой1=base, слой2=accent, слой3=ink).
- **Выход:** `front_<palette_id>.png` 300 DPI, готовые скормиться в `build_cover.py`.
- **Инвариант (главный критерий):** при фиксированной палитре пиксели зависят ТОЛЬКО
  от неё; геометрия идентична между вариантами (проверяемо: дифф формы при равной
  палитре = 0).

## Связи

- Кормит `engine/build_cover.py` (передний PNG на вход цепочки).
- Пересекается с `palette` из `niche.yaml` (STRUCTURE.md) — палитры могут браться оттуда.
- Перекраска ≠ наложение текста; текст по-прежнему делает `text_layout.py` поверх.
