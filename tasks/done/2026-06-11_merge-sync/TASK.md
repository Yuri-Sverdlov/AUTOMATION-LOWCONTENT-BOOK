## Контекст

Шёл merge удалённой ветки (2-й компьютер, сессии s9–s10) в локальную (s8). Конфликты были
в двух канонических файлах — **архитектор их уже разрешил вручную**:
- `CONTEXT.md` — оставлена версия s9–s10 (ревизия, приоритеты) + дописана ссылка на
  `tasks/backlog/2026-06_title-pipeline.md`.
- `tasks/TASK.md` — этот файл (разрешён).

Остальные файлы слились автоматически. Маркеры конфликта (`<<<<<<<`, `=======`, `>>>>>>>`)
в этих двух файлах убраны. Нужно завершить merge-коммит и запушить. **Никаких `--force`.**

## Шаг 0 — убедиться, что маркеров не осталось

```
git diff --check
grep -rn "<<<<<<<\|>>>>>>>\|=======" CONTEXT.md tasks/TASK.md
```
Если grep что-то находит (кроме безобидных `===` в обычном тексте) — СТОП, отчёт.
Ожидаемо: маркеров нет.

## Шаг 1 — добавить разрешённые файлы

```
git add CONTEXT.md tasks/TASK.md
git status
```
Должны быть помечены как разрешённые (resolved). Прочие конфликтные пути на этом этапе
не ожидаются (их слил git сам).

## Шаг 2 — завершить merge-коммит

```
git commit --no-edit
```
(сообщение слияния по умолчанию — оставь как есть).

## Шаг 3 — push

```
git push origin main
```

## Шаг 4 — проверка

```
git status -sb        # ожидаем: вровень с origin/main, дерево чисто
git log --oneline --graph -8
```

1. Маркеров конфликта в `CONTEXT.md`/`tasks/TASK.md` нет (`git diff --check` чисто). ✅/❌
2. Merge-коммит создан (впиши хэш); в нём видно слияние веток (`--graph`). ✅/❌

4. Финальный `git status -sb` = чисто, up to date with origin/main. ✅/❌
5. `git log --oneline -8` (приведи). ✅/❌

## Границы / СТОП-условия

- **НЕ** `git push --force`, **НЕ** rebase, **НЕ** трогать чужие ветки/историю.
- Если всплыли ЕЩЁ конфликтные файлы помимо двух разрешённых — СТОП, перечисли в REPORT,
  не разрешай канонические сам.
- Ошибка авторизации на push → СТОП, отчёт «нужен доступ к GitHub».
